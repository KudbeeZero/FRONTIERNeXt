## 1. THE BIG QUESTION

> *“Do all the parcels have to populate every single time? Are there major issues
> if we get the entire planet minted out?”*

**Answer: No — and your architecture already solves this correctly.**

Here’s what actually happens, based on the live code:

### Positions are NEVER stored — they’re computed deterministically

The 21,000 parcel positions are generated by a **Fibonacci sphere algorithm** that
runs identically on both client and server. `generateFibonacciSphere(21000)` produces
the exact same lat/lng for every plot, every time, on every machine. There is a hard
contract enforced in the code:

```
client/src/lib/globe/globeUtils.ts:
  "MUST produce the same lat/lng positions as server/sphereUtils.ts"
```

Both use the same golden angle (`Math.PI * (3 - Math.sqrt(5))`) and the same
`POLAR_EXCLUSION_LAT = 75`. This means:

- **Zero position data crosses the network.** The client computes all 21,000 positions
  locally in a `useMemo` on load.
- **Zero position data is stored in the DB.** Positions are math, not records.
- **Adding the millionth player changes nothing about geometry.**

### What IS stored: only ownership + state

The DB `land_parcels` table stores one row per plot, but the rows only carry
**mutable state** — `ownerId`, `biome`, `activeBattleId`, `isSubdivided`, resource
levels. Positions are derived from `plotId` at render time.

### The seeder writes 21k rows ONCE, in batches

`server/storage/seeder.ts` seeds parcels in **500-row batches** inside a transaction,
guarded by `game_meta.initialized`. This runs exactly once on first boot. After that,
`initialize()` is idempotent — it checks the flag and returns immediately.

**This is the right design. Do not change it.**

### So what’s the rendering cost of a full planet?

|Scenario                 |Rendering Cost           |Why                                   |
|-------------------------|-------------------------|--------------------------------------|
|Empty world (0 owned)    |1 InstancedMesh draw call|All 21k tiles in one buffer           |
|Half-minted (10.5k owned)|1 InstancedMesh draw call|Same — color is per-instance attribute|
|Fully minted (21k owned) |1 InstancedMesh draw call|**No change**                         |

**InstancedMesh means the GPU draws all 21,000 tiles in a single call regardless of
ownership.** A fully-minted planet costs exactly the same as an empty one. This is the
single most important architectural decision in the whole globe, and it’s already done
correctly.

## 2. HOW THE GLOBE RENDERS

### File map (the actual structure)

```
client/src/components/game/
  PlanetGlobe.tsx              — top-level R3F <Canvas> + scene composition
  globe/
    GlobeTerrain.tsx           — the planet sphere mesh + texture
    GlobeParcels.tsx           — InstancedMesh of 21k tiles (THE CORE, 473 lines)
    GlobeAtmosphere.tsx        — atmospheric glow shell
    StarField.tsx              — background starfield
    GlobeEvents.tsx            — battle arcs, mining pulses
    GlobeHUD.tsx               — 2D overlay (labels, readouts)
    GlobeEventOverlays.tsx     — orbital zones, satellites
  hooks/
    useGlobeCamera.ts          — fly-to, orbit, zoom controls

client/src/lib/globe/
    globeConstants.ts          — all colors, sizes, radii, speeds
    globeUtils.ts              — fibonacci sphere, lat/lng→vec3, color logic
    globeTypes.ts              — TS interfaces
```

### The render pipeline (GlobeParcels.tsx)

```
1. generateFibonacciSphere(21000)        → 21k {lat,lng} computed once (useMemo)
2. plotPositions3D: Float32Array         → flat xyz buffer for nearest-neighbor
3. InstancedMesh (fill) + InstancedMesh (border)
4. Per-instance: position matrix + color attribute
5. plotVisualFingerprint                 → only re-paints when ownership/battle/
                                            subdivision actually changes
6. useFrame loop                          → pulse animation for battles + selection
7. nearestPlot(x,y,z)                     → O(n) click/hover hit detection
```

### Key strengths already in place

- **Fingerprint-based repainting.** The mesh only rebuilds its color buffer when
  `plotVisualFingerprint` changes — a sorted string of only the plots that have an
  owner, battle, or subdivision. Idle frames cost nothing.
- **InstancedMesh.** One draw call for 21k tiles.
- **Deterministic positions.** Client/server parity, no network position payload.
- **Size variants.** `SIZE_VARIANTS` array breaks up visual monotony from plotId hash.

### Current weaknesses (what we’ll fix)

- Lighting is too dark (Section 6)
- Color system has a logic gap for the “I own this” case (Section 5)
- No animated ownership border (Section 6)
- No fog of war / scan layer (Section 7)
- Hit detection is O(n) per interaction — fine at 21k, but could use a spatial index
  if sub-parcels explode the count (Section 4)

## 3. WHERE EVERYTHING IS SAVED

### The storage split (this is the philosophy you asked about)

|Data                   |Where                                     |Why                                 |
|-----------------------|------------------------------------------|------------------------------------|
|Parcel positions       |**Nowhere — computed**                    |Deterministic Fibonacci math        |
|Parcel ownership/state |**Neon PostgreSQL** (`land_parcels`)      |Persistent world truth              |
|Sub-parcels            |**Neon PostgreSQL** (`sub_parcels`)       |Created lazily on subdivide         |
|Live game state, events|**Upstash Redis**                         |Fast ephemeral reads, battle replays|
|Land NFT ownership     |**Algorand chain** (ASA)                  |On-chain proof of ownership         |
|NFT metadata           |**PUBLIC_BASE_URL + chain**               |ARC-3 asset URL baked at mint       |
|Globe colors, sizes    |**Client constants** (`globeConstants.ts`)|Pure presentation, no DB            |

### The principle: minimize DB reliance, maximize determinism

You said *“we don’t want to completely rely on databases.”* The codebase already
honors this:

- **Positions** = math (no DB)
- **Colors** = constants (no DB)
- **Biome assignment** = deterministic from plotId (no DB lookup needed for render)
- **DB only stores mutable truth** = who owns what, what’s being built, active battles

This is exactly right for an Algorand-first game. The chain is the ownership source of
truth; the DB is the fast-read mirror; the client computes everything visual.

## 4. SCALING ISSUES

### Issue 1 — Sub-parcels can multiply the instance count

A fully-minted planet is 21,000 macro-plots. But if every plot subdivides into a 3×3
grid, that’s **189,000 sub-tiles**. The current `MAX_SUB_TILES = 9 * 500 = 4,500` cap
means only 500 macro-plots can show sub-grids at once.

**Recommendation:** Sub-parcels should only render when the camera is zoomed in past a
threshold (LOD — level of detail). At planet-wide zoom, show macro-plots only. As the
camera flies toward a region, swap in the sub-parcel InstancedMesh for the visible plots.

```typescript
// Pseudo-architecture for sub-parcel LOD
const showSubParcels = cameraDistance < SUB_PARCEL_LOD_THRESHOLD;
const visibleSubParcels = useMemo(() =>
  showSubParcels
    ? subParcels.filter(sp => isInViewFrustum(sp.parentPlotId, camera))
    : [],
  [showSubParcels, subParcels, cameraFrustumKey]
);
```

### Issue 2 — O(n) nearest-neighbor on every click

`nearestPlot()` loops all 21,000 positions per click. At 21k this is ~0.2ms — fine.
But if you ever raycast against sub-parcels too, build a simple **spatial hash grid**
keyed by lat/lng buckets so hit detection stays O(1).

### Issue 3 — Seeding 21k rows on a cold Neon free tier

The 500-row batched seed is good, but Neon’s free tier can be slow on cold starts.
**Upgrade to Neon Launch plan before mainnet** (your LUT already flags this).

### Issue 4 — Fingerprint string grows with ownership

`plotVisualFingerprint` builds a sorted string of all owned/battling/subdivided plots.
At full mint that’s a 21k-entry string rebuilt on every parcel data change. Consider a
**numeric hash** (XOR rolling hash of plotId+state) instead of a sorted string join to
keep it O(n) without the string allocation.

### Issue 5 — No issue with the planet being “minted out”

To be direct: **there is no rendering catastrophe from a fully-minted planet.** The
InstancedMesh handles it. The real scaling pressure is sub-parcels and DB write
throughput during high-traffic mint events — both solvable with LOD and Neon scaling.

## 5. COLOR SYSTEM

### Why you “can never get the color system to work”

Here’s the actual `getPlotColor` logic from `globeUtils.ts`:

```typescript
export function getPlotColor(parcel, currentPlayerId): THREE.Color {
  if (!parcel) return new THREE.Color(0x1a2a3a);          // dark fallback
  if (!parcel.ownerId) {
    const biomeCol = BIOME_COLORS[parcel.biome];
    return biomeCol ? biomeCol.clone() : new THREE.Color(0x1a2a3a);
  }
  if (currentPlayerId && parcel.ownerId === currentPlayerId)
    return COLOR_PLAYER.clone();                           // your mint-green
  return COLOR_ENEMY.clone();                              // enemy orange-red
}
```

**The logic is actually correct.** The problem is one of these three things:

### Root cause A — `currentPlayerId` is null at render time

If the player session hasn’t resolved when the globe first paints, `currentPlayerId`
is `null`, so **your own parcels render as enemy orange-red** until a re-render fires.
The fingerprint may not change (ownership didn’t change), so it never re-paints.

**Fix:** Add `currentPlayerId` to the `plotVisualFingerprint` dependency so the mesh
re-paints when the session resolves:

```typescript
const plotVisualFingerprint = useMemo(() => {
  return currentPlayerId + "|" + parcels   // ← prefix with player id
    .filter(p => p.ownerId || p.activeBattleId || p.isSubdivided)
    .map(p => `${p.plotId}:${p.ownerId ?? ""}:${p.activeBattleId ?? ""}:${Number(!!p.isSubdivided)}`)
    .sort()
    .join("|");
}, [parcels, currentPlayerId]);             // ← add dependency
```

### Root cause B — biome colors wash out against dark terrain

The unowned biome colors are neon, but the terrain underneath is dark and the fill
tiles are small (`FILL_SIZE = radius * 0.022`). At planet zoom they’re nearly invisible.

**Fix:** Add emissive material on the fill InstancedMesh so biome colors glow rather
than reflect (Section 6).

### Root cause C — no contrast between “mine” and “selected”

`COLOR_PLAYER` (mint-green) and `COLOR_SELECTED` (gold) are distinct, but with dark
lighting they read similarly. The animated ownership border (Section 6) solves this by
making ownership a *motion* signal, not just a color signal.

## 6. GLOBE VISUAL OVERHAUL

This is the “it’s OK but really dark” fix. Five targeted upgrades.

### 6.1 — Lighting rig

The globe is dark because it likely has weak ambient + one directional light. Replace
with a three-point rig + emissive tiles:

```typescript
// In PlanetGlobe.tsx scene
<ambientLight intensity={0.4} color="#4a6fa5" />          {/* cool fill */}
<directionalLight position={[5, 3, 5]} intensity={1.2} color="#fff4e6" /> {/* key */}
<directionalLight position={[-5, -2, -3]} intensity={0.5} color="#6a8fd5" /> {/* rim */}
<pointLight position={[0, 0, 8]} intensity={0.6} color="#ffffff" />  {/* camera-side */}
```

### 6.2 — Emissive parcel tiles

Make biome and ownership colors *glow* so they’re visible at planet zoom:

```typescript
// Fill mesh material
<meshStandardMaterial
  emissive={...}              // set per-instance via emissiveIntensity attribute
  emissiveIntensity={0.6}
  toneMapped={false}          // let neon colors stay saturated
  transparent
  opacity={0.85}
/>
```

### 6.3 — Animated ownership border (the pulsing effect you described)

For plots owned by the current player, render a **pulsing border ring** using a
shader-driven emissive border instance. The pulse uses `useFrame` time:

```typescript
useFrame((state) => {
  const t = state.clock.elapsedTime;
  const pulse = 0.5 + 0.5 * Math.sin(t * 2.5);   // 0..1 breathing
  // For each owned-by-me border instance, set emissiveIntensity = 0.4 + pulse * 0.6
  ownedBorderMesh.material.emissiveIntensity = 0.4 + pulse * 0.6;
});
```

A clean approach: a **third InstancedMesh** holding only the current player’s owned
plots, rendered as a slightly larger ring with an animated emissive pulse. This
separates “ownership glow” from the static fill/border meshes so it animates cheaply.

### 6.4 — Player color customization (sliders)

Let players pick their territory color. Store it client-side + optionally on the
player row:

```typescript
// New: player color preference
interface PlayerVisualPrefs {
  territoryColor: string;     // hex, default "#00ffaa"
  borderStyle: 'pulse' | 'solid' | 'glow';
  enemyColor: string;         // hex, default "#ff4400"
}
```

Drive `COLOR_PLAYER` from this pref instead of the hardcoded constant. Sliders in a
settings panel → HSL pickers → update a Zustand/context store → fingerprint includes
the color → mesh re-paints. **No DB required** if stored in localStorage; optional DB
column `players.territory_color` for cross-device persistence.

### 6.5 — Selection clarity

When a plot is selected, lift it slightly off the sphere surface (increase its
position matrix radius by ~2%) AND apply the gold color AND a static bright border.
Motion + color + lift = unmistakable.

## 7. FOG OF WAR AND OBSERVER

### 7.1 — Fog of war layer

Fog of war is a **visibility mask** over the parcel colors. Enemies/unscanned regions
render dimmed until revealed.

```typescript
// Visibility state per plot (client-computed, not stored)
enum PlotVisibility {
  HIDDEN   = 0,   // fog — render dark grey, no owner color
  SCANNED  = 1,   // revealed once — render at 50% with last-known state
  VISIBLE  = 2,   // live — full color, real-time updates
}
```

Visibility is derived from:

- Plots adjacent to your owned plots → VISIBLE
- Plots you’ve scanned (scan action) → SCANNED for N turns
- Everything else → HIDDEN

**Rendering:** add a per-instance `visibility` attribute. In the fragment shader (or
via emissiveIntensity), multiply color by `0.15` for HIDDEN, `0.5` for SCANNED, `1.0`
for VISIBLE. This is a pure visual layer — the DB doesn’t need to know what you can see.

### 7.2 — Scan mechanic

A scan reveals a radius of plots around a target for a number of turns:

```typescript
interface ScanAction {
  centerPlotId: number;
  radiusPlots: number;        // how many nearest plots revealed
  durationTurns: number;
  cost: { ascend: number };   // burn $ASCEND to scan
}
```

Scans cost $ASCEND (ties into your token economy). Reveal radius and duration scale
with scan tier. The revealed set is computed via the existing `nearestPlot` machinery
extended to “nearest N plots.”

### 7.3 — The Observer Mechanic (faster-than-light look-back)

This is your most ambitious idea. Let me ground it in something implementable.

**The physics intuition:** if an observer moves faster than light to a distant point
and looks back, they see the past — light that left earlier. Translated to game terms:

> The further out the observer’s camera pulls, the further *back in time* the world
> state they observe. Zooming out = looking into the past.

**Implementation as a deterministic world-state replay:**

```typescript
interface ObserverState {
  cameraDistance: number;       // GLOBE_RADIUS * N
  observedTurn: number;         // which historical turn you're seeing
}

// The mapping: camera distance → time offset
function observedTurnFromDistance(dist: number, currentTurn: number): number {
  const lightLag = Math.floor((dist - GLOBE_RADIUS) / LIGHT_LAG_PER_UNIT);
  return Math.max(0, currentTurn - lightLag);
}
```

When zoomed way out, the globe renders a **historical snapshot** — ownership and
battles as they were `lightLag` turns ago. Zoom in and you “catch up” to the present
as the light delay shrinks.

**Where the historical state comes from:** Upstash Redis already stores battle replays
and world events. Extend this to periodic **world-state snapshots** (every N turns,
store a compact ownership diff). The observer reads the snapshot matching `observedTurn`.

This makes the relativistic idea a real, deterministic mechanic: distance = latency =
historical view. It’s also genuinely novel and on-brand for your “dimensional observer”
roadmap item.

**Branching observers (the multi-dimensional angle):** different camera *angles* at the
same distance could seed different world branches (your existing roadmap note). Seed an
RNG with `(observerAngle, territory, burnHistory)` and render a divergent “what-if”
overlay. Keep this as a Phase 3+ experimental layer — it’s the visionary hook, not the
MVP.

## 8. SUBPARCEL ARCHETYPE SYSTEM

> Your idea: *“creating almost an archetype system for the sub plots — resource center,
> armory, economic, energy — and within each archetype, different buildings.”*

**Good news: the schema is already scaffolded for this.** The `sub_parcels` table has:

- `archetype` (`"resource"|"trade"|"fortress"|"energy"|null`)
- `archetypeLevel` (0=unassigned, 1-3)
- `energyAlignment` (`"helios"|"aegis"|"nexus"|null`)
- `improvements` (jsonb array — buildings live here)

So we extend, not rebuild. Here’s the full design.

### 8.1 — The four archetypes

Each sub-parcel (1 of 9 in a macro-plot’s 3×3 grid) is assigned an archetype. The
archetype determines which **buildings** can be constructed on it.

```
┌──────────────────────────────────────────────────────────────────┐
│  ARCHETYPE          THEME              CORE BENEFIT                │
├──────────────────────────────────────────────────────────────────┤
│  🏭 RESOURCE        Mining & refining   Boost mineral yield        │

```

This is the depth layer. Each archetype offers a tree of buildings:

### 8.3 — The energy alignment sub-system

`energyAlignment` (`helios`/`aegis`/`nexus`) adds a strategic layer to energy
sub-parcels — three power doctrines:

```
helios → offensive energy (powers weapons, air/space force)
aegis  → defensive energy (powers shields, defense grids)
nexus  → economic energy (powers refineries, trade, passive yield)
```

A player chooses an alignment when they build their first energy structure, creating a
build-order identity. This is real strategic depth from one existing column.

### 8.4 — How buildings affect the game (the wiring)

Buildings live in the `improvements` jsonb array. Their effects must flow into:

|Effect type          |Flows into              |File to wire                              |
|---------------------|------------------------|------------------------------------------|
|Resource yield       |Mining tick calculation |`server/storage/db.ts` resource accrual   |
|Trade bonuses        |Trade Station fee/speed |trade order resolution                    |
|Energy cost reduction|Upgrade/action costs    |action cost calculation                   |
|Weapon/defense tiers |Battle engine inputs    |`BattleInput` construction (NOT tuning.ts)|
|Air/space units      |New unit types in battle|battle engine extension                   |

**Critical rule (from your standing instructions):** building effects feed into
`BattleInput` as *inputs* — they do NOT modify `tuning.ts` constants. The engine stays
pure; buildings change what you feed it.

### 8.5 — Sub-parcel rendering (the 3×3 grid)

`GlobeParcels.tsx` already has a `SubParcelOverlay` — a 3×3 sub-tile grid for
subdivided macro-plots. Extend each sub-tile’s color to reflect its archetype:

```typescript
const ARCHETYPE_COLORS = {
```

Apply only at zoomed-in LOD (Section 4). At planet zoom, the macro-plot shows
`COLOR_SUBDIVIDED` (dark blue tint) to signal “this one is subdivided — zoom in.”

## 9. FRONTEND WIRING AUDIT

> *“All of the buttons are wired, the menus are working.”*

Here’s the systematic audit approach. Every interactive element must trace cleanly:

```
Button click → handler → action hook → API call → server route → 
storage method → DB/chain write → response → query invalidation → 
re-fetch → state update → fingerprint change → mesh re-paint
```

### Audit checklist per feature

|Feature         |Button/Menu          |Hook                  |Route                             |Status to verify             |
|----------------|---------------------|----------------------|----------------------------------|-----------------------------|
|Buy parcel      |LandSheet “Purchase” |useBlockchainActions  |POST /api/actions/purchase        |Wallet sign → ASA → DB       |
|Attack          |LandSheet “Attack”   |useBlockchainActions  |POST /api/actions/attack          |Source select → battle engine|
|Subdivide       |LandSheet “Subdivide”|(verify exists)       |POST /api/subparcels/create       |Creates 9 rows               |
|Assign archetype|SubParcel panel      |(NEW — needs building)|POST /api/subparcels/:id/archetype|Sets archetype col           |
|Build structure |Archetype panel      |(NEW)                 |POST /api/subparcels/:id/build    |Appends to improvements      |
|Scan            |Globe action bar     |(NEW)                 |POST /api/actions/scan            |Burns ASCEND, reveals        |
|Trade           |TradeStation         |useBlockchainActions  |POST /api/trade/*                 |Atomic order book            |
|Color prefs     |Settings panel       |(NEW — client only)   |localStorage                      |No server                    |

### The wiring rule when you change the globe

You said: *“if we’re changing anything then there has to be server.”* Here’s the
discipline — **classify every change first:**

```
PURE VISUAL (no server):     colors, lighting, borders, fog rendering, LOD
                             → client only, no route, no DB

STATE CHANGE (needs server):  ownership, buildings, archetype, scans, battles
                             → route + storage method + query invalidation
```

If a change only affects how something *looks*, it never touches the server. If it
changes *what is true about the world*, it always does. This is the line that keeps the
architecture clean.

## 11. UPDATE ORDER

> *“I don’t know if where you update first or what you do.”*

This is the dependency order. **Always update in this sequence** so nothing breaks
downstream:

```
1. SCHEMA FIRST (if state shape changes)
   server/db-schema.ts + shared schema
   → archetype already exists; new fields (scan state, color pref) go here
   → if migration needed, flag it; prefer additive nullable columns

2. SERVER STORAGE METHODS
   server/storage/db.ts
   → the methods that read/write the new state
   → keep battle/transaction structure intact

3. SERVER ROUTES
   server/routes.ts
   → expose the storage methods as API endpoints
   → add to the existing route structure, don't restructure

4. SHARED TYPES
   → so client and server agree on the shape

5. CLIENT HOOKS
   client/src/hooks/
   → useBlockchainActions etc — the API call + query invalidation

6. CLIENT STATE / QUERY
   → TanStack Query keys, invalidation on mutation

7. GLOBE CONSTANTS (if visual)
   client/src/lib/globe/globeConstants.ts
   → colors, sizes — pure presentation

8. GLOBE RENDERING
   client/src/components/game/globe/
   → consume new state, update fingerprint deps, re-paint

9. UI PANELS
   → buttons/menus that trigger the hooks from step 5
```

**The golden rule:** data shape flows down (schema → server → client → render).
Visual-only changes start at step 7 and never touch 1–6. When you told me “if we change
the globe there has to be server” — that’s only true when the *state* changes. Pure
visual globe changes start at step 7.
